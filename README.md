# DQ Time Series Service

DQ Time Series Service - MVP-сервис для мониторинга качества данных в PostgreSQL на основе временных рядов. Он помогает быстро увидеть, что в данных изменилось не так: резко просел поток строк, выросла доля `NULL`, сломалось распределение сумм, пропали уникальные значения или появилась другая аномалия.

Проект ориентирован на аналитические команды, которым нужно не просто запускать разовые SQL-проверки, а отслеживать поведение метрик во времени.

## Какую проблему решает

Классические data-quality проверки часто отвечают на вопрос “прошли ли данные фиксированное правило”. Но многие реальные сбои выглядят иначе:

- вчера 10 000 строк в час было нормально, а сегодня 2 000 - уже инцидент;
- средний чек не нарушил жесткий лимит, но резко ушел от привычного диапазона;
- доля пустых значений растет постепенно, и это видно только в динамике;
- количество уникальных клиентов или статусов изменилось относительно обычного поведения.

DQ Time Series Service превращает такие проверки в временные ряды: сервис считает агрегаты по новым данным, строит прогноз, а затем отмечает отклонения как аномалии.

## Основной пользовательский сценарий

1. Пользователь создает подключение к PostgreSQL.
2. Проверяет, что сервис может подключиться к базе.
3. Создает мониторинг для таблицы: выбирает schema/table, checkpoint-колонку, cron-расписание, набор метрик и модель.
4. Запускает мониторинг вручную или включает автоматический scheduler.
5. Сервис считает агрегаты по новым строкам и сохраняет точки временных рядов.
6. После накопления истории сервис строит прогноз при помощи выбранной пользователем модели и отмечает выбросы.
7. Пользователь смотрит дэшборд, график временных рядов и карточки аномалий.

## Метрики

Сервис поддерживает табличные и колоночные метрики.

Табличные:

- `row_count` - количество строк в новом batch;
- `empty_batch` - признак отсутствия новых строк.

Числовые:

- `min`, `max`, `avg`, `sum`, `stddev`;
- `zero_ratio`, `negative_ratio`;
- `null_ratio`.

Категориальные и строковые:

- `distinct_count`;
- `unique_ratio`;
- `empty_ratio`;
- `avg_length`.

## Модели поиска аномалий

Модель задается в `model_config` монитора.

| Модель | Когда полезна |
| --- | --- |
| `rolling` | Базовый rolling window: простой и понятный baseline. |
| `robust_z` | Устойчивее к выбросам за счет median/MAD. |
| `exp_smoothing` | Подходит для плавно меняющихся рядов. |
| `seasonal_naive` | Учитывает повторяющийся сезонный паттерн. |
| `quantile_boosting` | Нелинейная модель с quantile-диапазоном. |
| `random_forest` | Нелинейные зависимости по лагам ряда. |
| `isolation_forest` | Поиск необычных точек и локальных паттернов. |

Если `model` не указан, используется `rolling`. Если имя модели неизвестно или параметр имеет некорректный тип/диапазон, запуск мониторинга завершится ошибкой, которую можно увидеть в истории запусков.

Общие параметры:

- `window` - сколько последних точек брать для обучения или статистики;
- `k` - множитель ширины диапазона для статистических моделей;
- `min_spread` - минимальная ширина диапазона около прогноза;
- `lower_quantile` и `upper_quantile` - квантили для границ диапазона в ML-моделях;
- `lags` - сколько предыдущих точек использовать как признаки для ML-моделей;
- `random_state` - фиксирует повторяемость ML-результата.

Перед построением прогноза сервис должен накопить минимум `MIN_SERIES_POINTS` исторических точек. До этого точки сохраняются, но модельные границы могут быть пустыми, а аномалии будут находиться только по `static_rules`.

Примеры конфигураций:

`rolling` - хороший дефолт для ровных метрик без сложной сезонности:

```json
{
  "model": "rolling",
  "window": 30,
  "k": 3
}
```

`robust_z` - полезен, когда в истории уже были единичные выбросы и обычное среднее слишком чувствительно:

```json
{
  "model": "robust_z",
  "window": 60,
  "threshold": 3.5
}
```

`exp_smoothing` - подходит для плавного тренда:

```json
{
  "model": "exp_smoothing",
  "alpha": 0.35,
  "k": 3
}
```

`seasonal_naive` - используйте для повторяющегося цикла, например `season_length: 24` для почасовых данных с суточной сезонностью:

```json
{
  "model": "seasonal_naive",
  "season_length": 24,
  "tolerance": 0.2,
  "min_spread": 1
}
```

`quantile_boosting` - нелинейный прогноз по лагам с диапазоном по residual-квантилям:

```json
{
  "model": "quantile_boosting",
  "window": 240,
  "lags": 24,
  "lower_quantile": 0.05,
  "upper_quantile": 0.95,
  "n_estimators": 100,
  "learning_rate": 0.05,
  "max_depth": 3,
  "random_state": 42
}
```

`random_forest` - более устойчивый нелинейный baseline для рядов с лаговыми зависимостями:

```json
{
  "model": "random_forest",
  "window": 240,
  "lags": 24,
  "lower_quantile": 0.05,
  "upper_quantile": 0.95,
  "n_estimators": 200,
  "max_depth": null,
  "min_samples_leaf": 2,
  "random_state": 42
}
```

`isolation_forest` - ищет необычный текущий паттерн напрямую. Границы на графике строятся для визуальной подсказки, но решение об аномалии принимает isolation score:

```json
{
  "model": "isolation_forest",
  "window": 240,
  "lags": 24,
  "contamination": 0.03,
  "n_estimators": 100,
  "random_state": 42
}
```

Для небольшого количества истории и простых DQ-метрик обычно начинайте с `rolling` или `robust_z`. Для почасовых/дневных циклов берите `seasonal_naive`. ML-модели имеют смысл, когда по каждому ряду уже накоплено достаточно точек, а метрика зависит от предыдущих значений нелинейно.

## Расписание мониторингов

Автоматический запуск задается cron-выражением в поле `schedule_cron`. Используется классический формат из пяти полей:

```text
minute hour day month weekday
```

Примеры:

| Cron | Значение |
| --- | --- |
| `*/5 * * * *` | Каждые 5 минут. |
| `0 * * * *` | В начале каждого часа. |
| `0 9 * * 1-5` | В 09:00 по будням. |
| `30 2 * * *` | Каждый день в 02:30. |

Поле `timezone` определяет часовой пояс, в котором интерпретируется cron-выражение.


## Быстрый запуск

Нужны Docker и Docker Compose.

```bash
docker compose up -d --build
```

После запуска:

- UI: http://localhost:5173
- Swagger UI: http://localhost:8000/docs
- Health: http://localhost:8000/api/v1/health
- Ready: http://localhost:8000/api/v1/ready

По умолчанию используются локальные настройки из `.env.example`. Для подключения собственного источника создайте PostgreSQL connection в UI или через API.

## Конфигурация окружения

Для локального запуска достаточно `.env.example`. Для собственного окружения создайте `.env` и задайте реальные значения.

```bash
cp .env.example .env
```

| Переменная | Назначение |
| --- | --- |
| `ADMIN_USERNAME` | Логин администратора. |
| `ADMIN_PASSWORD` | Пароль администратора. |
| `ADMIN_TOKEN` | Токен для заголовка `X-Admin-Token`. |
| `ENCRYPTION_KEY` | Ключ для шифрования паролей подключений. |
| `DATABASE_URL` | DSN внутренней БД сервиса. |
| `MIN_SERIES_POINTS` | Минимум исторических точек перед прогнозом. |
| `CORS_ORIGINS` | Разрешенные origins для frontend. |

Если `ADMIN_TOKEN=change-me`, backend работает в локальном открытом режиме и не требует admin-токен. Для любого закрытого окружения обязательно замените `ADMIN_PASSWORD`, `ADMIN_TOKEN` и `ENCRYPTION_KEY`.
